Real case:The product team uses "automatic acceptance mode" to let AI build Vim mode independently, and 70% of the code is completed by AI.

How to do it in HappyCapy:

1. Make sure you start with a clean Git state

2. Give detailed but abstract tasks

"Help me implement a user authentication system:
- Support email + password login
- JWT token authentication
- Include registration, login, and password reset
- Write unit tests

3. Check the progress regularly and take over when 80% completed

  • HappyCapy Automatically run tests and fix errors
  • You focus on the last 20% of your core business logic

Best practices:

  • ✅ Suitable for: edge functions, experimental prototypes, unfamiliar technology stacks
  • ❌ Not suitable for: core business logic, security-critical code
  • 💾 Submit Git checkpoints at each stage to facilitate rollback

💡 Advice for developers

1. Establish a self-validation loop

"When implementing this feature:
- Run tests automatically after coding
- Run lint checks for code style
- If tests fail, analyze root cause and fix proactively
- Finally return code that passes all tests"

2. Develop intuition for task classification

  • asynchronous autonomous: Edge functionality, prototypes, unfamiliar technologies
  • Synchronous supervision: Core business logic, security critical, production environment

3. Write clear and detailed prompts

  • Give explicit file paths when repositories are similarly named
  • State your coding style preference (e.g. "program functionally, avoid classes")
  • Provide context (e.g. "This project uses TypeScript + React + Zustand")

4. Use multiple instances to work in parallel

  • HappyCapy supports multiple tabs, each tab is an independent session
  • One tab handles the frontend, the other handles the backend
  • Maintain full context for each session, making switching stress-free

5. Treat refactoring tasks like a "slot machine"

  • Submit current status (git commit）
  • Let HappyCapy work autonomously for 30 minutes
  • If you are satisfied with the result, accept it; if you are not satisfied, start over (git reset --hard）
  • Starting over often has a higher success rate than correcting mistakes
